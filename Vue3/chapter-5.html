01 · 学习目标

学完这一章你要能做到

  • 解释 selectDatazoneDoc 与 selectDatazoneFile 的区别：前者选文档（返回 docid/url/type），后者选微盘文件（返回 fileId/url）。
  • 说清鉴权 → 调接口的完整链路：注册身份 → 拿签名 → 调专区接口 → 后端用返回值做分析。

阅读路径：背景包 → 全链路地图 → 核心判断 → 关键概念 → 做关卡题 → 生成报告。

这是把前四章串起来的落地章。前四章你学了 Vue3 选项式、SFC、computed、组件化；这一章回答：「在企微 H5 项目里，这些 Vue 知识怎么和企微 JSAPI 结合，把『文档检索能力需求』真正做出来」。

02 · 零基础背景包

读这章前先知道的 4 件事

背景①企微 H5 不是普通网页，它跑在企微客户端里

本项目员工端 H5 是在企业微信 App 里打开的（通过企微管家员工端入口）。这意味着它能调用企微独有的能力（选文档、选文件、转发等），这些能力普通浏览器没有，要通过 JSAPI 才能用。

读这章需要它：否则你不知道为什么要学一套"企微专属 API"。
背景②"调企微接口"不是直接调，要先证明身份

企微不让随便一个网页就调它的接口。你的 H5 必须先通过 身份注册（ww.register），证明「我属于这个企业、有权用这个应用」。注册成功后才能调 JSAPI。这就像进机房要先刷工牌。

读这章需要它：所有 JSAPI 调用的前提都是"先 register 成功"。
背景③签名是后端给的，前端不自己算

身份注册要一份"签名"，但这个签名必须由后端算——因为算签名要用到 jsapi_ticket，而 ticket 是用企业的 access_token 换的，access_token 绝不能暴露到前端。前端的活是：请求后端拿签名 → 喂给 ww.register。

读这章需要它：理解为什么 ww.register 的签名是回调函数（异步拿后端的）。
背景④"文档检索"要用户先选文档，再让后端分析

本需求「数据与智能专区—文档检索能力」的核心流程：用户在专区里选文档/文件 → 前端拿到 docid/fileId → 传给后端 → 后端调通用模型分析 → 返回结果给前端渲染。选文档用的就是 selectDatazoneDoc/File 两个接口。

读这章需要它：这是整个需求的业务主线，本章所有内容都围绕它。
03 · 这章在回答什么

表面 vs 深层问题

表层：怎么调企微 JSAPI、怎么选文档、怎么渲染消息。

深层：从「用户点开应用」到「看到分析结果」，前端要和企微客户端、自家后端三方怎么协作？身份怎么证明？数据怎么流动？

只记一个问题：在企微 H5 里，前端调一个 JSAPI（比如选文档），从注册身份到拿到结果，完整经过哪几步？每步谁负责？

带着这个问题读「04 结构地图」，那条全链路就是答案。
04 · 结构地图

一条全链路：从打开应用到看到结果

本章不是零散知识点的堆砌，而是一条完整的业务链路。下面 6 步就是用户从点开应用到看到分析结果的全过程，每一步对应一个角色（前端/后端/企微客户端）。

1进入应用｜用户从企微工作台点开应用 → 经中转页 ?type=xxxx 重定向到目标 H5 页面。
2前端启动｜Vue 应用挂载（mounted），引入企微 jssdk（npm @wecom/jssdk 或 CDN）。
3身份注册 ⭐｜调 ww.register，传 corpId/agentId/jsApiList + 两个签名回调。签名回调里去请求自家后端拿签名。
4调用接口｜register 成功后，调 ww.selectDatazoneDoc 或 ww.selectDatazoneFile，弹出选择器。
5拿到结果｜用户选完，回调返回 docid/fileId+url → 前端把这些传给自家后端。
6渲染结果｜后端调通用模型分析完，返回内容 → 前端用 Vue（v-html + DOMPurify）渲染富文本。
节点关系：步骤 3（身份注册）是步骤 4（调接口）的硬前提——不 register 直接调 JSAPI 会报无权限。这是贯穿全章的因果主线。
05 · 核心判断

6 条核心判断（每条都带原文依据）

事实① ww.register 统一完成企业身份 + 应用身份注册，是新版主推方式。

在新版 wecom-jssdk 中，统一使用 ww.register 进行身份注册，取代旧版 wx.config + wx.agentConfig 两段式。一个 ww.register 调用同时搞定企业签名和应用签名。

原文依据：企微《ww.register》(94313)——「在新版的 wecom-jssdk.js 中，统一使用 ww.register 进行身份注册」+《开始使用》(90514)。
事实② 所有 JS 接口只能在企微应用的可信域名下调用，且可信域名必须 ICP 备案。

这是硬性约束。本地开发常用 localhost，但 localhost 不在可信域名里，签名会失败。必须配测试域名（在企微后台备案）或用企微内部环境。

原文依据：企微《开始使用》(90514)——「所有的JS接口只能在企业微信应用的可信域名下调用，且可信域名必须有ICP备案」。
事实③ ww.register 的签名通过 getConfigSignature/getAgentConfigSignature 回调异步获取，由后端算。

签名不是前端写死的，也不是前端算的。ww.register 接收两个回调函数，回调被调用时带着 url 参数，前端用这个 url 去请求自家后端，后端算好签名返回，前端再传给 sdk。所以是"异步拿签名"。

原文依据：企微《ww.register》(94313)——getConfigSignature / getAgentConfigSignature 回调参数说明。
事实④ selectDatazoneDoc 选「文档」，返回 docid+url+type；selectDatazoneFile 选「微盘文件」，返回 fileId+url。

这是本需求的两个核心入口接口。两者是兄弟关系，调用前提都是"必须先 ww.register + 在应用文档/文件存档范围内"。type 字段标识文档类型（3=文档/4=表格/5=收集表/6=幻灯片等）。

原文依据：企微《数据与智能专区文档选择》(101499) + 《文件选择》(101544)——调用前提节均明确"必须先 ww.register"。
判断⑤ 应用身份权限 > 企业身份权限：应用身份能调所有企业接口 + 应用专属接口。

ww.register 如果只传 corpId（不传 agentId），只注册企业身份，能调的接口有限。传了 agentId 才注册应用身份，能调 selectDatazoneDoc/File 这种应用专属接口。本需求必须注册应用身份。

原文依据：企微《开始使用》(90514)——应用身份权限与企业身份权限的层级关系。
判断⑥ 企微应用消息不支持直接下发 HTML，复杂富文本靠"消息卡片 + H5 落地页"组合。

企微消息类型是 textcard/template_card/markdown 等，不支持 HTML 富文本。要让用户看到富排版，常规做法是：消息卡片只放摘要+跳转链接，用户点进去到 H5 页面，前端用 v-html（配合 DOMPurify 防 XSS）渲染真正的富文本。

原文依据：企微《发送应用消息》(90236) 消息类型不支持 HTML；Vue《模板语法》v-html 的 XSS 风险。⚠️ 老田原话称"渲染组件"，但未给专门文档，此处为社区通行做法，具体以员工端项目为准。
06 · 关键概念拆解

6 个关键概念

① ww.register（身份注册）

定义：企微新版 jssdk 的统一身份注册接口，一次调用完成企业签名 + 应用签名。

ww.register 关键参数
ww.register({
  corpId: 'ww1234567890abcdef',        // 必填，企业ID
  agentId: '1000001',                   // 应用ID（传了才注册应用身份）
  jsApiList: ['selectDatazoneDoc'],     // 要用的接口名
  getConfigSignature(params) {          // 企业签名回调（异步找后端拿）
    return fetch('/api/js-signature', {
      method:'POST', body: JSON.stringify(params)
    }).then(r => r.json())
    // 返回 { signature, nonceStr, timestamp }
  },
  getAgentConfigSignature(params) {     // 应用签名回调
    return fetch('/api/agent-signature', {
      method:'POST', body: JSON.stringify(params)
    }).then(r => r.json())
  }
})
不要误解成：签名是前端算的。签名必须后端算（要用到不能暴露的 access_token/ticket），前端只负责"请求后端拿回来喂给 sdk"。

② getConfigSignature / getAgentConfigSignature（签名回调）

定义：ww.register 的两个回调函数，sdk 需要签名时会调用它们，传入当前页 url，期望返回签名对象。

工作流：sdk 调回调(url) → 前端拿 url 请求后端 → 后端用 jsapi_ticket + url + nonce + timestamp 算 SHA1 → 返回 {signature, nonceStr, timestamp} → 前端 resolve 给 sdk。

原文依据：《ww.register》——两回调的参数与返回结构。

③ ww.checkJsApi（探测接口可用性）

定义：检测当前客户端是否支持某些 JS 接口。注意它是"探测"，不是鉴权——鉴权是 ww.register 的活。

ww.checkJsApi({
  jsApiList: ['selectDatazoneDoc', 'selectDatazoneFile'],
  success(res) { console.log(res.checkResult) }
  // { selectDatazoneDoc: 'ok', selectDatazoneFile: 'ok' }
})
不要误解成：checkJsApi 通过了就能调。它只测"客户端支不支持"，调之前还得 register 过身份。

④ ww.selectDatazoneDoc（选文档）

定义：弹出文档选择器，用户选完返回 docList（每项含 docid + url + type）。

返回字段说明
docid文档 ID，传给后端做分析
url文档访问 URL
type3=文档/4=表格/5=收集表/6=幻灯片/7=思维导图/8=流程图/10=智能表格/11=智能文档
原文依据：《数据与智能专区文档选择》(101499)。
调用前提：必须先 ww.register + 在应用文档存档范围内 + 有专区文档存档权限，否则报 83000 no permission。

⑤ ww.selectDatazoneFile（选微盘文件）

定义：弹出微盘文件选择器，用户选完返回 fileList（每项含 fileId + url）。与 selectDatazoneDoc 是兄弟接口。

对比selectDatazoneDocselectDatazoneFile
选什么文档（doc）微盘文件
返回docid + url + typefileId + url
权限文档存档权限文件存档权限
原文依据：《数据与智能专区文件选择》(101544)。

⑥ v-html + DOMPurify（富文本渲染）

定义：后端返回的富文本（HTML 字符串）要在 Vue 里渲染，用 v-html。但 v-html 直接渲染有 XSS 风险，必须先用 DOMPurify 净化。

选项式：渲染后端返回的富文本
<!-- template -->
<div v-html="safeContent"></div>

<!-- script（选项式）-->
import DOMPurify from 'dompurify'
export default {
  data() { return { rawContent: '' } },
  computed: {
    safeContent() {                    // 用 computed 净化
      return DOMPurify.sanitize(this.rawContent)
    }
  },
  methods: {
    async loadResult(docid) {
      const res = await fetch('/api/analyze', {
        method:'POST', body: JSON.stringify({ docid })
      })
      const data = await res.json()
      this.rawContent = data.content   // 原始HTML存进 data
    }
  }
}
不要误解成：v-html 直接渲染就行。后端内容也可能被注入恶意脚本，无论来源都要净化。配合 computed 净化（带缓存，第四章学的）正合适。

09 · 误解与边界

4 个最容易踩的坑

误解①：ww.register 和旧版 wx.config 一样，随便用哪个

真相：新版 wecom-jssdk 统一用 ww.register，旧版 config+agentConfig 是迁移对象。本项目用新版 sdk（@wecom/jssdk），就走 ww.register。别在新 sdk 里还写 wx.config。

原文依据：《ww.register》——「统一使用 ww.register 进行身份注册」。

误解②：签名前端自己算就行

真相：算签名要用 jsapi_ticket，ticket 用 access_token 换，access_token 绝不能暴露到前端。签名必须后端算，前端通过 getConfigSignature 回调异步拿。

误解③：selectDatazoneDoc 注册完就能调

真相：register 只是第一步。还要满足：当前用户在应用可见范围内 + 在文档存档范围内 + 应用有专区文档存档权限（灰度内测）。任一不满足都报 83000 no permission。

原文依据：《文档选择》(101499) 调用前提 + 错误码 83000。

误解④：v-html 渲染后端内容很安全

真相：v-html 直接渲染任何来源的 HTML 都有 XSS 风险，后端内容也可能被注入。必须用 DOMPurify 净化后再 v-html。配合 computed 净化（带缓存）最合适。

边界：本章覆盖「ww.register 鉴权 + 专区两接口 + v-html 渲染 + 全链路」。后端签名算法细节（SHA1 拼接）、专区存档权限开通流程、中转页 ?type=xxxx 具体映射规则、员工端项目实际请求封装，属于项目内部实现，需问居易/土豆或在员工端代码里看，本章只给通用框架。

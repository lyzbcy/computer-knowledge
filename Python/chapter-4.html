第四章 · 错误与异常 —— 让程序遇到意外时不崩溃

语法错误 vs 异常 ｜ try/except ｜ else/finally ｜ raise ｜ 自定义异常 ｜ 含 … 道互动关卡

01 · 学习目标

学完这一章你要能做到

  • 分清两类错误：语法错误（SyntaxError，解析阶段）vs 异常（Exception，运行时），知道它们何时发生。
  • 用 try/except 捕获异常：知道 try 至少要配一个 except，会指定异常类型（如 except ValueError）。
  • 理解 else 和 finally：else 是没异常时才执行；finally 是无论如何都执行。
  • 用 raise 主动抛异常：会抛出指定异常，知道参数必须是异常实例或类。
  • 自定义异常：通过继承 Exception 定义自己的异常类。
阅读路径：背景包 → 结构地图 → 核心判断 → 关键概念 → 做关卡题 → 生成报告。
本章定位

这是 Python 学习地图的第四章。前三章你学了语法、控制流、数据结构，能写出有用的程序。但程序运行时难免遇到意外（除零、文件不存在、用户输错）——本章教你用 try/except 让程序优雅地处理意外而不崩溃。内容全部基于 Python 3.12 官方教程第 6 章《错误和异常》。

原文依据：Python 3.12 官方教程第 6 章《错误和异常》——涵盖语法错误、异常、异常处理、抛出异常、用户自定义异常、清理行为（finally）。
02 · 零基础背景包

读这章前先知道的 4 件事

背景①程序会出错，错误分两大类

Python 的错误至少分两种：语法错误（代码写得不对，解释器都解析不了）和异常（语法对、但运行时出问题，如除零）。区分它们是处理错误的第一步。

读这章需要它：否则你不知道什么能用 try 捕获、什么不能。
背景②崩溃 vs 优雅处理

没有异常处理时，遇到异常程序会直接崩溃并打印一长串错误信息（Traceback）。用 try/except 可以捕获异常并决定怎么处理——比如提示用户重新输入，而不是整个程序挂掉。

读这章需要它：理解 try/except 存在的意义。
背景③异常是一种"对象"

Python 里的异常不只是个错误信息，它是一种对象（类实例），有类型（如 ValueError、TypeError）和参数。你可以按类型捕获不同的异常，分别处理。

读这章需要它：理解为什么 except ValueError 能只抓这一种异常。
背景④有些清理工作必须做

打开文件、连数据库后，无论是否出错都要关闭，否则资源泄漏。Python 的 finally 子句专门保证这种"无论如何都执行"的清理逻辑。

读这章需要它：理解 finally 为什么不能省。
03 · 这章在回答什么

表面 vs 深层问题

表层：try/except/finally/raise 怎么写。

深层：Python 怎么区分"写错代码"和"运行时意外"？try 的工作流程是什么？为什么 finally 一定执行？为什么自定义异常要继承 Exception？

只记一个问题：程序运行时遇到意外，怎样捕获并处理异常而不崩溃？try/except/else/finally/raise 各自的作用和执行时机是什么？

带着这个问题读「05 核心判断」和「06 关键概念」，它们都在回答它。
04 · 结构地图

错误与异常四层递进

官方教程第 6 章按"识别 → 处理 → 主动抛 → 自定义"的顺序展开：先分清两类错误，再学用 try/except 处理，然后学主动 raise，最后学自定义异常类。

1识别错误｜语法错误（SyntaxError，解析时）vs 异常（Exception，运行时）。
2处理异常 ⭐｜try/except 捕获，可指定异常类型；else 无异常时执行；finally 一定执行。
3主动抛｜raise 强制抛出指定异常，参数须是异常实例或类。
4自定义异常｜继承 Exception 创建自己的异常类。
节点关系：先识别（分清语法错误和异常），再处理（try/except），能处理就能主动抛（raise），能抛就能自定义。其中 try/except 是核心。
05 · 核心判断

6 条核心判断（每条带官方原文依据）

事实① 错误分两类：语法错误（SyntaxError，解析时）和异常（Exception，运行时）。

语法错误是解析阶段发现的（如少了冒号、括号不匹配），报 SyntaxError。异常是语法正确、运行时才暴露的问题（如 10/0 的 ZeroDivisionError）。

原文依据：Python 3.12 官方教程第 6 章——「错误可（至少）被分为两种：语法错误和异常。即使语句或表达式在语法上是正确的，但在尝试执行时，它仍可能会引发错误。在执行时检测到的错误被称为异常。」
事实② try 至少要配一个 except：try 子句执行时若发生异常且类型匹配，则执行对应 except。

try 子句中没有异常则跳过 except；发生异常且类型和 except 后的异常匹配，则执行该 except。try 不能单独存在，必须至少跟一个 except 或 finally。

原文依据：Python 3.12 官方教程第 6 章《异常的处理》——「如果没有异常发生，则跳过 except 子句……如果异常的类型和 except 关键字后面的异常匹配，则执行 except 子句。」
事实③ else 子句在 try 没有抛出任何异常时才执行。

else 属于 try 语句，只有 try 子句没发生异常时才执行。它常用来放"成功了才要做的事"，让 try 子句保持精简。

原文依据：Python 3.12 官方教程第 6 章——「try 语句有另一个可选的 else 子句……else 子句在 try 子句没有发生异常时执行。」
事实④ finally 子句总是在离开 try 语句时被执行，无论是否发生异常。

finally 一定会执行——有异常、没异常、异常被捕获、异常没被捕获，甚至 return/break，finally 都会跑。它专用于释放资源（关文件、断连接）。

原文依据：Python 3.12 官方教程第 6 章《定义清理行为》——「一个 finally 子句总是在离开 try 语句时会被执行的，无论是发生了异常没有发生。」
事实⑤ raise 语句主动抛出指定异常，参数必须是异常实例或异常类（派生自 Exception）。

raise NameError('HiThere') 抛出一个 NameError。raise ValueError 是 raise ValueError() 的简写。参数必须是异常类或实例，不能是别的。

原文依据：Python 3.12 官方教程第 6 章《抛出异常》——「raise 语句允许程序员强制抛出一个指定的异常……这个参数必须是一个异常实例或者是一个异常类（派生自 Exception 的类）。」
判断⑥ except 可以指定异常类型（如 except ValueError），只捕获匹配的异常；自定义异常应继承 Exception。

except ValueError 只抓 ValueError，别的异常照样崩溃。要自定义异常，新建一个类继承 Exception（直接或间接）。异常名通常以 "Error" 结尾。

原文依据：Python 3.12 官方教程第 6 章《用户自定义异常》——「程序可以通过创建新的异常类来命名自己的异常。异常通常应该直接或间接地从 Exception 类派生。」
06 · 关键概念拆解

6 个关键概念

① 语法错误（SyntaxError）

定义：解析阶段（代码还没运行）就发现的错误，如少冒号、括号不匹配。最常见的入门错误。

while True print('Hello world')
#   File "", line 1
#     while True print('Hello world')
#                ^^^^
# SyntaxError: invalid syntax
关键：语法错误不能用 try 捕获（代码根本没运行）。错误箭头指向的位置不一定是真正要改的地方。

② 异常（Exception）

定义：语法正确、但运行时检测到的错误。每个异常有类型名。

常见异常类型
10 * (1/0)   # ZeroDivisionError: division by zero
4 + spam*3   # NameError: name 'spam' is not defined
'2' + 2      # TypeError: can only concatenate str to str
错误信息最后一行告诉你异常类型——这是用 except 按类型捕获的依据。

③ try / except 捕获异常

定义：把可能出错的代码放 try 里，用 except 处理捕获到的异常。可指定异常类型。

while True:
    try:
        x = int(input("请输入数字: "))
        break
    except ValueError:
        print("不是有效数字，再试一次！")

# 可绑定异常实例
except OSError as err:
    print("OS error: {0}".format(err))
一个 try 可有多个 except，分别处理不同类型；最多只有一个会被执行。

④ else / finally

定义：else 在 try 没异常时执行；finally 无论如何都执行。

子句何时执行用途
elsetry 没抛异常时放"成功才做的事"
finally无论有无异常都执行释放资源（关文件/断连）
try:
    raise KeyboardInterrupt
finally:
    print('Goodbye, world!')
# 无论是否抛异常，finally 都执行
不要误解成：finally"看情况执行"。错！它一定执行。

⑤ raise 主动抛异常

定义：raise 强制抛出一个指定异常。参数必须是异常实例或类。

raise NameError('HiThere')   # 抛带消息的异常
raise ValueError             # raise ValueError() 的简写

# 捕获后再抛出
try:
    raise NameError('HiThere')
except NameError:
    print('飞过一个异常！')
    raise                    # 重新抛出当前异常
单独写 raise（无参数）在 except 块里会重新抛出刚捕获的异常。

⑥ 自定义异常（继承 Exception）

定义：定义新类继承 Exception（直接或间接），命名自己的异常。通常以 "Error" 结尾。

class Error(Exception):
    """本模块异常的基类。"""
    pass

class InputError(Error):
    """输入错误引发的异常。"""
    def __init__(self, expression, message):
        self.expression = expression
        self.message = message
不要误解成：自定义异常随便继承什么都行。它应继承 Exception（直接或间接），否则 except 抓不到、语义也不对。

09 · 误解与边界

4 个最容易踩的坑

误解①：finally 看情况执行，有异常就不执行

真相：finally 一定执行——有异常、没异常、异常被捕获、异常未被捕获，它都跑。这正是它用于资源清理的原因。

原文依据：第6章《定义清理行为》——「一个 finally 子句总是在离开 try 语句时会被执行的，无论是发生了异常没有发生。」

误解②：语法错误能用 try 捕获

真相：语法错误在解析阶段就发现了，代码根本没开始运行，try 捕获不到它。只有运行时异常才能用 try 捕获。

误解③：except 不写类型就什么都抓不到 / 或什么都能抓

真相：不写类型的 except: 能捕获几乎所有异常（对，但不推荐——它会连 KeyboardInterrupt 这种用户中断也吞掉，还掩盖真正的 bug）。最好指定具体类型。

误解④：自定义异常随便继承什么都行

真相：自定义异常应直接或间接继承 Exception。继承别的类（如 object）就不是"异常"，except 也抓不到。异常名通常以 "Error" 结尾。

原文依据：第6章《用户自定义异常》——「异常通常应该直接或间接地从 Exception 类派生。」
边界：本章覆盖语法错误/异常/try/except/else/finally/raise/自定义异常。未深入：异常链（raise...from）、预定义清理行为（with 语句）、异常层次体系——这些在官方教程同章但属进阶。
10 · 背景知识缺口

要继续学还差什么

现在就该练（阻碍写代码）：

  • 写一个 try/except—— 捕获 input 转数字时的 ValueError，让用户重试。
  • 试 finally—— 在 try 里 raise 一个异常，看 finally 是否照样执行。

可以晚点补：

  • with 语句—— 文件、锁等资源的预定义清理，比 finally 更简洁（with open(f) as ...）。
  • 异常链—— raise ... from ... 在处理异常时抛另一个并保留原始信息。
  • 异常层次体系—— BaseException / Exception / 各具体异常的继承关系。
  • 面向对象（类）—— 自定义异常、更复杂的程序结构都依赖类，是进阶必修。

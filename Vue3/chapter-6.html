02 · 项目背景包

读这章前先知道的 4 件事

背景①这个需求要做什么（一句话）

企微开放了「数据智能专区—文档存档能力」（本质是检索能力）。本项目要在「企微 AI 管家」（PC 管理后台的智能体）里，让用户能选择企微文档/微盘文件，把内容取出来传给智能体分析。

原文依据：需求会议——「数据智能专区跟之前的专区会话类似……文档存档本质是检索能力……只有专区能通过关键词或指定id检索文档内容。」
背景②先做需求二，不是需求一

项目有两个需求点。芳芳明确：会先做需求二（企微AI管家文档/微盘分析增强），再做需求一（文档管家skill）。本章聚焦需求二，因为它先要上线（7月24号目标）。

原文依据：需求会议——「会先做需求二，再做需求一（skill）。把两个需求拆开。先保第二个需求能上线。」
背景③为什么拿不到文档原文，只能拿"总结"

企微不开放原始文件内容（专区数据原则上我们拿不到完整原文）。必须走「通用模型」，模型返回的是总结的结果（文字片段），不是原文。前端拿这个文字片段回显 + 传智能体。

原文依据：需求会议——「企微不开放原始文件内容，必须走通用模型才能拿到结果……返回的是总结的结果（文字片段）。」
背景④前端只做交互层，重活在后端

调专区通用模型（创建任务+获取结果）是后端的活，前端不碰。前端要的是后端把这些能力封装成 HTTP 接口给你调。你的边界：拼二维码、调JSAPI拿id、轮询后端接口、回显、传参。

原文依据：接口清单——「前端只做交互层。调专区通用模型是后端的活，前端要的是后端封装成HTTP接口。」随风：「前端给我doc_id → 我调通用模型 → 打包给智能体。」

04 · 前端任务全景

你要做的 7 件事（按用户操作顺序）

这是本章核心。下面 7 个任务就是用户从看到入口到看到分析结果的全过程，每一步标注「用到前五章哪个知识」。

1加输入框入口｜AI管家输入框下加「企微文档/微盘」下拉。用 v-if+授权判断控制显隐。→ 用第三章 v-if、第六章授权拦截
2拼二维码 ⭐核心｜用户选了文档类型，前端拼一个带 session_id 的链接，生成二维码展示。PC无法直调组件所以要扫码。→ 项目专属，新学
3调JSAPI拿id｜手机企微扫码后唤起原生选择器，选完返回 doc_id/file_id+url+type。→ 用第五章 ww.register+selectDatazoneDoc/File
4轮询拿结果 ⭐｜前端拿id传后端，轮询后端接口拿通用模型返回的"文档内容文字"。→ 项目专属（轮询机制），新学
5回显卡片｜拿到结果后，用卡片形式回显（图标+标题）。标题靠前端展示组件用URL展示。→ 用第四章组件化(props传卡片数据)
6发送传智能体｜点发送，把文档内容文字作为新参数传智能体（像现在传企业信息一样）。→ 用第二章 SFC 组织这部分逻辑
7授权拦截｜页面加载查后端"是否专区存档授权"，没授权隐藏入口或引导。→ 用第三章 computed+mounted
节点关系：任务2（拼二维码）和任务4（轮询）是项目独有的新知识，前五章没覆盖，本章重点讲。其余5项都是前五章知识的项目化应用。
05 · 核心判断

6 条核心判断（每条带项目原文依据）

事实① 二维码本质是前端拼的一个链接，核心作用是打开企微"选择文档/微盘"入口。

不是后端生成、不是企微给的。前端用 JS 拼一个 URL（带 session_id 对应当前对话），转成二维码图片展示。链接页本身不做任何内容，只负责唤起企微原生选择器。

原文依据：需求会议——「二维码本质是前端组装的一个链接……链接要带参session_id……前端做两个链接：一个跳文档、一个跳微盘。」
事实② PC 无法直调企微组件，所以要扫码；手机企微环境内可直接放链接。

这是为什么要扫码的根本原因。PC 管理后台跑在浏览器里，调不了企微客户端组件；用户得用手机企微扫二维码，跳进企微客户端才能唤起原生选择器。这是需求二（PC端）和需求一（skill，可能企微环境内）的区别。

原文依据：需求会议——「PC无法直调组件→扫码」「企微环境内打开可直接放链接，无需扫码；非企微环境展示二维码。」
事实③ 通用模型是异步 job，前端要轮询拿结果。

拿文档内容不是同步返回的。后端调专区模型是"创建任务+获取结果"两步，异步。前端传id后，要轮询后端接口直到拿到结果。（待确认：前端轮询 vs 后端推送 SSE/WebSocket——见待确认缺口）

原文依据：需求会议——「通用模型是异步的，是个job，必须拿到结果才算成功。」接口清单接口①待确认项。
判断④ 前端拿不到文档原文，只能拿到通用模型总结的"文字片段"。

企微不开放原始内容。回显卡片显示的是模型总结的文字，不是文档全文。这个文字片段也是传给智能体的内容（不是传链接，是传一堆文字）。

原文依据：需求会议——「企微不开放原始文件内容……返回总结的结果（文字片段）……把它作为附件形式给智能体。」文浩：「跟本地文件有点像，只是这个传一堆文字。」
判断⑤ 授权判断依据是"应用权限集"，不是 user_doc_ticket。

前端页面加载时要查后端"是否已专区存档授权"，没授权隐藏入口。判断依据是应用权限集（会话/文档/知识库三个独立权限），不是看 ticket 拿不拿得到——这点还没完全确认（见缺口）。

原文依据：需求会议——「是个应用权限。」随风：「ticket是否代表授权，还不太确认。」
事实⑥ 前端要和后端对齐 5 个接口。

不是 5 个前端自己写的接口，而是要和后端确认怎么调的 5 个能力：①拿文档内容（核心）②是否授权判断 ③智能体新参数 ④JSAPI选文件（联调）⑤标题展示组件。其中①②③是后端HTTP接口，④⑤是企微能力但要和后端对齐返回值怎么用。

原文依据：前后端接口对齐清单——「接口全景·一共5个。」
06 · 关键概念拆解

6 个项目专属概念

① 二维码链接（前端拼）

定义：前端用 JS 拼一个 URL，带 session_id 参数，转成二维码图片。用户手机企微扫码后，跳转到企微原生选择组件。

拼链接 + 生成二维码（选项式伪代码）
// 选项式：methods 里拼链接
methods: {
  genQrLink(type) {                 // type: 'doc' 或 'file'
    const sid = this.sessionId       // 当前对话id（已有）
    const base = 'https://域名/redirect'
    return `${base}?session_id=${sid}&type=${type}`
  }
},
mounted() {
  this.docQrUrl = this.genQrLink('doc')
  this.fileQrUrl = this.genQrLink('file')
  // 然后用二维码库（如 qrcode.js）把 url 转成图片
}
不要误解成：二维码是后端生成的。会议反复确认——前端拼，客户端行为，不需要服务端。

② 轮询拿结果

定义：前端传 doc_id 给后端后，因为通用模型异步，前端要反复调一个查询接口，直到拿到结果。

轮询的选项式骨架（待和后端确认形态）
methods: {
  async pollDocContent(docId) {
    const taskId = await this.createTask(docId)  // 调"创建"接口
    const timer = setInterval(async () => {       // 反复查
      const res = await this.queryTask(taskId)    // 调"查询"接口
      if (res.status === 'done') {
        clearInterval(timer)
        this.docContent = res.content             // 拿到文字片段
        this.showCard = true                       // 回显卡片
      }
    }, 2000)                                       // 每2秒查一次
    // ⚠️ 待确认：轮询key用task_id还是session_id？超时多久？
  }
}
用到的Vue知识：setInterval（JS基础）、async/await（异步）、data 存状态、v-if 控制卡片显隐。待确认：是前端轮询两段式（创建+查询），还是后端封装成一接口内部轮询。

③ 文档标题展示组件

定义：JSAPI 选完返回的有 URL 但没有标题。要拿到标题，得用企微的"前端展示组件"（用 URL 渲染出标题）。

回显卡片需要：图标（用 type 对应）+ 标题（用展示组件拿）+ 可能的状态。这部分要确认展示组件具体怎么调（找居易/土豆）。

原文依据：需求会议——「返回doc_list（含doc_id）、还有URL和type，但没有标题，标题要再通过前端展示组件（用URL展示）。」

④ session_id（对话标识）

定义：当前对话的 id。拼二维码链接时要带上，作用是——手机端选完文档后，结果能回传到对应的 PC 端对话。

这是跨设备关联的关键：PC 生成带 sid 的二维码 → 手机扫码选文档 → 结果带着 sid 回来 → PC 端这个对话收到结果。

⑤ 智能体新参数（传文档内容）

定义：发送提问时，把拿到的文档内容文字，作为新增的一个参数传给智能体，跟现在传"企业信息/页面数据"一样。

和本地文件的区别：本地文件传的是链接；企微文档/微盘传的是一堆文字（模型总结的内容）。参数名、结构要和后端统一（待确认：叫 doc_content？结构是字符串还是 {title,content,type}？）。

原文依据：需求会议——「我们会加一个参数告诉他是客户选择的企微文档对应的内容。」接口清单接口③。

⑥ 前端任务 ↔ 前五章知识映射

前端任务用到的章节知识
加入口/显隐控制第三章 v-if + computed（授权状态）
组织页面逻辑第二章 SFC（.vue 三段式，选项式）
回显卡片第四章 组件化（DocCard 组件 + props 传数据）
调JSAPI拿id第五章 ww.register + selectDatazoneDoc/File
渲染分析结果第五章 v-html + DOMPurify
列表渲染多个文档第三章 v-for + key
结论：前五章没有白学，每一章都在这个项目里有具体落点。新增要学的只有"拼二维码"和"轮询机制"两个项目专属技能。

09 · 误解与边界

4 个最容易踩的坑

误解①：前端要调专区通用模型

真相：调专区模型（创建任务+获取结果）是后端的活。前端只调后端封装好的 HTTP 接口。你的边界：拼二维码、调JSAPI、轮询、回显、传参。

原文依据：接口清单——「前端只做交互层。调专区通用模型是后端的活。」

误解②：二维码是后端生成的

真相：二维码是前端拼的链接转成的。前端用 JS 拼 URL（带 session_id），用二维码库转图片。不需要后端参与。

误解③：选完文档能拿到原文

真相：企微不开放原始内容。只能拿到通用模型总结的文字片段。回显和传智能体用的都是这个总结，不是原文。

误解④：JSAPI 返回的有标题

真相：selectDatazoneDoc/File 返回的有 doc_id/file_id + url + type，没有标题。标题要另外用企微"前端展示组件"拿 URL 渲染。

边界：本章只覆盖需求二（PC端企微AI管家）的前端部分。需求一（文档管家skill）后做，且 skill 本质是后端封装的开放接口，前端相关只有"环境判断→链接or二维码"一小块，不在本章重点。客户画像页交互调整（输入框引导文案、提示条、手动触发分析）属于细节交互，按设计稿做。
10 · 待确认缺口（开工前必须对齐）

这些没定死，开不了工

和后端（随风）对齐的 5 个接口问题：

  • 接口①轮询形态：前端两段式（创建+查询）还是后端封装成一接口内部轮询？轮询 key 用 task_id 还是 session_id？超时多久？P0 必须定
  • 接口②授权判断依据：用应用权限集还是 user_doc_ticket？给字段还是给接口？挂哪个现有接口？P0
  • 接口③参数命名：文档内容参数叫什么？结构是字符串还是对象？P1
  • 接口①入参结构：doc_id 数组怎么传？要不要带 type 区分文档/微盘？P1
  • 接口①出参结构：返回纯文本还是结构化（标题/正文/摘要）？回显卡片用什么字段？P1

找同事确认的：

  • 标题展示组件怎么调——用URL渲染标题的企微组件，找居易/土豆
  • 二维码这套流程有没有现成参考——会议提到"之前测过这套流程"，找诺诺/娟娟
  • 员工端项目请求封装/目录结构——看员工端现有代码（老田："参考员工端其它页面"）
  • 分支管理/提测/发布流程——团队工程规范，老田要求的"全流程"
建议下一步：拿着「前后端接口对齐清单.html」和随风过一遍 5 个接口（尤其接口①②），确认后本章的伪代码就能落地成真代码。在那之前，先在前端把不依赖后端的部分做出来：拼二维码、调JSAPI的骨架、回显卡片组件。

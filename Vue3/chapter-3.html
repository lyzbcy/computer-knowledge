第三章 · 计算属性、侦听器与条件/列表渲染

computed 缓存 ｜ v-if vs v-show ｜ v-for 与 key ｜ watch 副作用｜含 … 道互动关卡

01 · 学习目标

学完这一章你要能做到

  • 解释 computed 的缓存机制：为什么计算属性比"在模板里写表达式"或"调用方法"更优。
  • 区分 v-if 与 v-show：知道哪个会销毁重建、哪个只切 CSS，并说出何时选哪个。
  • 正确使用 v-for + :key：默写列表渲染语法，知道 key 必须唯一稳定、为什么用 index 当 key 有隐患。
  • 区分 computed 与 watch：派生值用 computed，副作用（异步、改 DOM）用 watch，不混用。
  • 能指出至少 2 个常见误解（如"v-if 和 v-show 一样""v-for 不写 key 也行"）。
阅读路径：背景包 → 结构地图 → 核心判断 → 关键概念 → 做关卡题 → 生成报告。
项目约束延续

本章所有示例仍用选项式 API：computed 是组件选项、watch 是组件选项，this 访问状态。禁用 computed(() => ...)、watchEffect() 这类组合式函数。

原文依据：《初心与使命》——「禁止使用"组合式"API」。
02 · 零基础背景包

读这章前先知道的 4 件事

背景①模板里能写表达式，但写多了会乱

第一章你知道 {{ message }} 能插值。其实 {{ books.length > 0 ? '有书' : '没书' }} 这种表达式也能写。但复杂逻辑塞进模板，页面会变得难读、难维护。计算属性就是为解决这个而生的。

读这章需要它：否则你不知道为什么要有 computed。
背景②有些逻辑是"算出一个值"，有些是"做一件事"

比如"根据生日算年龄"，结果是一个值——这是派生，适合 computed。比如"年龄变了，去发个请求"，过程是一个动作——这是副作用，适合 watch。区分这两类是本章的核心判断力。

读这章需要它：computed 和 watch 的选择全靠这条判断。
背景③"显示/隐藏"和"创建/销毁"是两回事

一个元素不显示，可以用 CSS 把它藏起来（display:none），它还在 DOM 里；也可以干脆不创建它，DOM 里压根没有。v-show 是前者，v-if 是后者。性能开销不同。

读这章需要它：v-if vs v-show 的取舍就是这两条路线的取舍。
背景④列表渲染要给 Vue 一个"身份证"

渲染一个数组列表时，Vue 内部要跟踪每个项的身份，以便数据变化时精准更新对应 DOM。这个"身份证"就是 key。没有 key，Vue 只能靠"位置"猜，容易出错；有了稳定唯一的 key，更新才准。

读这章需要它：v-for 的 key 不是可选项，是正确性的前提。
03 · 这章在回答什么

表面 vs 深层问题

表层：computed、watch、v-if、v-for 这些指令/选项怎么用。

深层：Vue 给你这么多种"响应式地处理数据"的工具，它们各自解决什么独特的问题？什么时候该用哪个、不该用哪个？

只记一个问题：computed / 方法 / watch 三者，面对同一份响应式数据，到底各自适合干什么？

带着这个问题读「05 核心判断」，你会发现它们分工明确、不可互相替代。
04 · 结构地图

四个工具，两类用途

本章四个工具可分成两大类：「派生值」类（computed）和「控制渲染/副作用」类（v-if/v-show/v-for/watch）。它们都建立在第一章的响应式之上。

1computed｜从响应式数据派生新值，带缓存，getter 无副作用。
2v-if / v-show ⭐条件渲染｜v-if 真销毁重建，v-show 只切 display。
3v-for + :key｜列表渲染，key 唯一稳定是正确更新的前提。
4watch｜数据变化时执行副作用（异步请求、改 DOM），不返回值。
节点关系：computed 和 watch 都"响应数据变化"，但 computed 出值、watch 做事；v-if 和 v-for 都控制"渲染什么"，但一个管条件、一个管循环，且两者不要写在同一元素上。
05 · 核心判断

6 条核心判断（每条都带原文依据）

事实① 计算属性的值会基于其响应式依赖被缓存。

这是 computed 区别于"方法"的根本。多次访问同一个计算属性，只要依赖没变，就返回缓存值，不重新计算；依赖变了才重算。

原文依据：Vue《计算属性》——「计算属性值会基于其响应式依赖被缓存」「一个计算属性仅会在其响应式依赖更新时才重新计算」。
事实② 同样逻辑写成"方法"，每次重渲染都会重新执行。

把计算逻辑写成一个 method 并在模板里调用，每次组件重渲染都会重新跑一遍函数，没有缓存。computed 有缓存，所以更高效。

原文依据：Vue《计算属性》——「方法调用总是会在重渲染发生时再次执行函数」（对比计算属性的缓存）。
判断③ computed 的 getter 不应有副作用。

计算属性的 getter 应是纯计算：不改变其他状态、不发异步请求、不操作 DOM。需要副作用时，用 watch。

原文依据：Vue《计算属性》最佳实践——「Getter 不应有副作用」「不要改变其他状态、在 getter 中做异步请求或者更改 DOM」。
事实④ v-if 是"真实条件渲染"——会销毁和重建；v-show 只切 display。

v-if 为 false 时，元素从 DOM 移除，事件监听器和子组件都销毁；为 true 时重建。v-show 则元素始终在 DOM，只是 display 切换。所以 v-if 切换开销大、v-show 初始开销大。

原文依据：Vue《条件渲染》——「v-if 是真实的按条件渲染……事件监听器和子组件都会被销毁与重建」「v-show 会在 DOM 渲染中保留该元素……仅切换了 display 的 CSS 属性」。
判断⑤ 频繁切换用 v-show，运行时很少变用 v-if。

这是 v-if 与 v-show 的取舍准则。v-if 有更高切换开销（销毁重建），v-show 有更高初始开销（都渲染）。按"切换频率"选。

原文依据：Vue《条件渲染》——「v-if 有更高的切换开销，而 v-show 有更高的初始渲染开销。如果需要频繁切换则使用 v-show 较好；如果在运行时绑定条件很少改变则 v-if 更合适」。
事实⑥ v-for 应提供唯一的 key，且 key 应是基础类型（字符串/数字），不要用 index。

key 让 Vue 跟踪每个列表项身份。用数组 index 当 key，在列表增删时会错位，导致状态串项。应使用每项稳定的唯一标识（如 id）。

原文依据：Vue《列表渲染》——「需要为每个元素提供一个唯一的 key attribute」「key 绑定的值期望是一个基础类型的值……不要用对象作为 v-for 的 key」。
06 · 关键概念拆解

6 个关键概念

① 计算属性 computed（选项式）

定义：组件的 computed 选项，用 getter 函数从响应式数据派生新值，带缓存。

选项式 computed 写法
export default {
  data() {
    return {
      author: { name: '张三', books: ['Vue 指南', 'JS 高级'] }
    }
  },
  computed: {
    hasBooks() {                      // getter，无参数
      return this.author.books.length > 0
    }
  }
}
<!-- 模板里当属性用，不加括号 -->
<p>{{ hasBooks ? '有书' : '没书' }}</p>
不要误解成：computed 要在模板里"调用"（加括号）。它当属性直接用，不用加括号。

② computed vs 方法

同样一段"算有没有书"的逻辑，写成 method 也能跑：

维度computedmethods
缓存✅ 有（依赖不变不重算）❌ 无（每次重渲染都重算）
模板用法{{ hasBooks }}{{ hasBooks() }}
适用派生值需要参数/事件处理
原文依据：Vue《计算属性》——「方法调用总是会在重渲染发生时再次执行函数」。

③ v-if / v-else-if / v-else

定义：根据表达式的真假，真实地渲染或不渲染一块内容。为 false 时元素不在 DOM。

<div v-if="type === 'A'">A</div>
<div v-else-if="type === 'B'">B</div>
<div v-else">其他</div>
v-if 是惰性的：初次为 false 时完全不渲染。

④ v-show（只切 display）

定义：根据表达式真假，切换元素的 display CSS 属性。元素始终在 DOM。

<div v-show="visible">我一直在 DOM，只是可能被 display:none</div>
不要误解成：v-show 比 v-if"高级"。它们用途不同：频繁切换选 v-show，条件少变选 v-if。

⑤ v-for + :key

定义：基于数组渲染列表。item in items 或 (item, index) in items。每个项要给唯一稳定的 :key。

<li v-for="book in books" :key="book.id">
  {{ book.title }}
</li>
不要误解成：用 index 当 key 没问题。列表会增删时，index 会错位导致状态串项。要用每项稳定的唯一 id。

⑥ 侦听器 watch（选项式）

定义：组件的 watch 选项，在被侦听的数据变化时执行副作用（如发请求、改 DOM、设定时器）。

选项式 watch 写法
export default {
  data() { return { question: '' } },
  watch: {
    // 方法名 = 被侦听的数据名
    question(newVal, oldVal) {
      if (newVal.includes('?')) {
        this.getAnswer()    // 副作用：发请求
      }
    }
  },
  methods: {
    getAnswer() { /* fetch ... */ }
  }
}
computed vs watch：要一个派生值 → computed；要在变化时做事 → watch。能用 computed 就别用 watch，因为 computed 声明式、有缓存、更不易出错。

09 · 误解与边界

4 个最容易踩的坑

误解①：v-if 和 v-show 一样，随便用

真相：v-if 会销毁/重建（切换开销大），v-show 只切 display（初始开销大）。频繁切换用 v-show，条件少变用 v-if。性能取舍不同。

原文依据：《条件渲染》——「v-if 有更高的切换开销，而 v-show 有更高的初始渲染开销」。

误解②：v-for 不写 key 也能跑，没事

真相：不写 key，Vue 用"位置"猜，列表增删时容易状态串项（比如输入框内容跑错行）。官方推荐任何可行的时候都给 key。

原文依据：《列表渲染》——「推荐在任何可行的时候为 v-for 提供一个 key attribute」。

误解③：computed 的 getter 里可以发请求

真相：getter 应是纯计算，不能有副作用（异步请求、改状态、改 DOM）。要副作用用 watch。官方明确反对在 getter 里做异步。

原文依据：《计算属性》最佳实践——「Getter 不应有副作用」「不要……在 getter 中做异步请求」。

误解④：computed 能做的事 watch 也能做，反过来也行

真相：分工不同。computed 出值（派生、缓存、声明式），watch 做事（副作用、异步、命令式）。能用 computed 就别用 watch——computed 更不易出错。

边界：本章覆盖 computed/v-if/v-show/v-for/watch 五个工具的核心用法与取舍。可写计算属性（getter+setter）、watch 的 deep/immediate/flush 细节选项、Vue 对数组的变更方法与替换方法细节，属于进阶，需要时再查官方文档。

第四章 · 组件化与 props/emit

组件复用 ｜ props 单向数据流 ｜ emit 子→父 ｜ emits 校验｜含 … 道互动关卡

01 · 学习目标

学完这一章你要能做到

  • 说清组件是什么：UI 的独立可复用单元，把界面拆成小块互相组合。
  • 默写选项式组件的 props 声明：数组式和对象式（带校验），并解释单向数据流。
  • 用 emit 实现子→父通信：子组件触发事件、父组件监听，并知道 emits 选项的作用。
  • 区分 props（父→子）与 emit（子→父）：知道数据为什么单向流动、为什么不直接改 props。
  • 能指出至少 2 个常见误解（如"子组件直接改 props""props 和 emit 两个方向随意用"）。
阅读路径：背景包 → 结构地图 → 核心判断 → 关键概念 → 做关卡题 → 生成报告。
项目约束延续

本章所有组件示例仍用选项式 API：props 是组件选项、emits 是组件选项，this.$emit 触发事件。禁用 defineProps、defineEmits 这类组合式宏。

原文依据：《初心与使命》——「禁止使用"组合式"API」。
02 · 零基础背景包

读这章前先知道的 4 件事

背景①页面写大了会失控，所以要拆

前三章你把所有逻辑、模板写在一个组件里。一旦页面复杂（几十个功能块），一个文件装不下、改一处怕牵连别处。组件化的核心动机就是把大页面拆成小块，每块独立维护、互相组合。

读这章需要它：否则你不知道为什么要发明"组件"这个概念。
背景②组件 = 一个可复用的"自定义元素"

组件本质是你自己造一个标签，比如 <TodoItem>，它在页面里就像 <div> 一样能用，但行为是你定义的。组件内部有自己的 data、methods、template，对外通过 props（进）和 emit（出）沟通。

读这章需要它：这是理解 props/emit 的前提——组件是有"接口"的黑盒。
背景③"数据归谁管"是组件化的核心问题

拆了组件后，状态（数据）该放在父组件还是子组件？这决定了数据怎么流动。Vue 的答案是：数据归一方管，通过 props 向下传、通过 emit 向上通知，不直接互相改。这叫单向数据流。

读这章需要它：props 和 emit 的设计就是为了维持这个单向流动。
背景④HTML 属性不区分大小写

HTML 里 data-id 和 dataId 是两个不同属性，且浏览器会把属性名转小写。所以在模板里写 props 时要用 kebab-case（短横线），而 JS 里声明用 camelCase（驼峰）。Vue 会自动转换。

读这章需要它：否则你会困惑"为什么 props 声明用驼峰、模板里写短横线"。
03 · 这章在回答什么

表面 vs 深层问题

表层：怎么定义组件、怎么传 props、怎么触发 emit。

深层：组件化后，数据在父子组件之间该怎么流动？为什么 Vue 强制单向数据流、严禁子组件改 props？这套设计解决了什么问题？

只记一个问题：父组件要把数据给子组件用 props，子组件要通知父组件用 emit——为什么不能反过来、为什么不能直接改？

带着这个问题读「05 核心判断」，尤其是判断②（单向数据流）。
04 · 结构地图

组件通信：一个"黑盒接口"模型

把组件想成一个黑盒：props 是入口（父→子）、emit 是出口（子→父）。父组件通过 props 把数据"喂"进子组件，子组件不直接改这些数据，而是通过 emit 事件"通知"父组件，由父组件决定怎么改。这就维持了清晰的数据流向。

1组件定义｜用选项对象定义一个可复用单元，有独立的 data/methods/template。
2props（父→子） ⭐入口｜父组件通过 props 把数据传给子组件；子组件声明要接收哪些。
3emit（子→父） ⭐出口｜子组件用 this.$emit 触发事件；父组件用 @事件名 监听。
4单向数据流｜数据从父流向子，子不直接改 props，要改就 emit 通知父改。
节点关系：props 和 emit 是一对方向相反的通道，共同构成父子通信。单向数据流是它们的设计准则，不是 props 或 emit 单独的属性。
05 · 核心判断

6 条核心判断（每条都带原文依据）

事实① 组件允许我们将 UI 划分为独立的、可重用的部分。

这是组件化的官方定义。每个组件有自己的模板、逻辑、样式，封装好后像一个自定义元素一样在父组件里使用。

原文依据：Vue《组件基础》——「组件允许我们将 UI 划分为独立的、可重用的部分」。
判断② 所有的 props 都遵循单向绑定原则：子组件严禁直接修改 props。

这是 props 最重要的一条铁律。父组件传下来的 props，子组件只能读、不能直接改。要"改"就得通过 emit 通知父组件去改。这样数据流向始终清晰（父→子），便于追踪和调试。

原文依据：Vue《Props》——「所有的 props 都遵循着单向绑定原则」「props 严禁被子组件更改」。
事实③ props 声明用 camelCase（JS），模板传递用 kebab-case（HTML）。

因为 HTML 属性不区分大小写且会转小写。JS 里声明 props: { postTitle: String }，模板里写 :post-title="..."。Vue 自动转换两者。

原文依据：Vue《Props》——camelCase 与 kebab-case 的转换说明。
事实④ props 可用对象式声明做校验：类型、必填、默认值、自定义校验。

数组式 props: ['title'] 只声明名字；对象式 props: { title: { type: String, required: true, default: '' } } 能加校验。校验失败时 Vue 在控制台警告（开发环境）。

原文依据：Vue《Props》——Prop 校验（type/required/default/validator）。
事实⑤ 子组件用 this.$emit 触发事件，父组件用 v-on（@）监听。

这是子→父通信的标准方式。子组件里 this.$emit('submit', data)，父组件模板里 <ChildComp @submit="handleSubmit" />。事件名建议用 kebab-case。

原文依据：Vue《组件事件》——触发与监听事件。
判断⑥ emits 选项可以声明并校验组件触发的事件。

和 props 校验类似，emits 选项声明组件会触发哪些事件，还能加校验函数。这既是文档（告诉使用者这个组件会 emit 什么），也能防止事件名拼写错误。

原文依据：Vue《组件事件》——emits 选项（数组式/对象式，对象式可带校验）。
06 · 关键概念拆解

6 个关键概念

① 组件（选项式定义）

定义：一个用选项对象描述的可复用 UI 单元。本项目里就是 .vue 文件导出的 export default { ... }。

子组件 TodoItem.vue（选项式）
export default {
  props: ['todo'],              // 接收父传来的 todo
  emits: ['remove'],            // 声明会触发 remove 事件
  methods: {
    onRemove() {
      this.$emit('remove', this.todo.id)   // 通知父组件
    }
  }
}
父组件用 <TodoItem :todo="item" @remove="handleRemove" /> 组合它。

② props（父→子的入口）

定义：组件的 props 选项，声明它要从父组件接收哪些数据。

声明方式写法能力
数组式props: ['title', 'likes']只声明名字
对象式props: { title: String }+ 类型
对象式完整{ title: { type: String, required: true, default: '' } }+ 必填/默认/校验
不要误解成：props 是"双向"的。它是单向入口，子组件不能直接改。

③ 单向数据流

定义：数据从父组件流向子组件（通过 props），子组件不直接改 props，要变更就 emit 通知父组件。

为什么这么设计：如果子组件能直接改 props，数据来源就模糊了——"这个值到底是父改的还是子改的？"调试会变成噩梦。单向流动让每次变更都可追溯。

原文依据：Vue《Props》——「所有的 props 都遵循着单向绑定原则」。
子组件想"改"props 怎么办？：两种合规做法——① 把 props 转成本地 data（data() { return { local: this.propVal } }）再改本地副本；② 直接 emit 事件让父改。

④ emit（子→父的出口）

定义：子组件用 this.$emit('事件名', 载荷) 触发一个自定义事件，父组件用 @事件名 监听。

<!-- 子组件内部 -->
this.$emit('update', newValue)

<!-- 父组件模板 -->
<ChildComp @update="handleUpdate" />
不要误解成：emit 是"调用父组件的方法"。它是发一个事件，父组件决定怎么响应（也可能不响应）。解耦的关键。

⑤ emits 选项（声明+校验事件）

定义：组件的 emits 选项，声明这个组件会触发哪些事件，类似 props 声明。

emits 两种写法
// 数组式：只声明
emits: ['update', 'remove']

// 对象式：带校验
emits: {
  update(payload) {
    return payload !== undefined   // 返回 false → 控制台警告
  }
}
emits 选项不是必须的，但官方推荐——它是组件的"接口文档"，也帮助 Vue 区分自定义事件和原生事件。

⑥ props vs emit（方向对比）

维度propsemit
方向父 → 子子 → 父
载体数据/值事件 + 载荷
子组件接收（只读）触发
父组件传递监听（@事件名）
校验选项propsemits
口诀：props 是"传进来"（只读），emit 是"喊一声"（父决定怎么办）。两者配合实现单向数据流。

09 · 误解与边界

4 个最容易踩的坑

误解①：子组件可以直接改 props

真相：props 严禁被子组件更改（单向数据流）。要"改"就转成本地 data 副本，或 emit 通知父组件。直接赋值会触发 Vue 警告，且数据来源混乱。

原文依据：《Props》——「props 严禁被子组件更改」。

误解②：props 和 emit 方向随意

真相：props 永远是父→子，emit 永远是子→父，方向固定。不能"用 props 把子组件的数据传给父"，那是 emit 的活。

误解③：模板里 props 用驼峰写

真相：HTML 属性会转小写，:postTitle 会被浏览器变成 :posttitle，匹配不上。模板里必须用 kebab-case（:post-title），JS 声明用 camelCase，Vue 自动转换。

误解④：emit 是"调用父组件的方法"

真相：emit 是发一个事件，不是直接调用。父组件监听并决定怎么响应（也可能不响应）。这是解耦的关键——子组件不需要知道父组件会做什么。

边界：本章覆盖组件基础、props（声明/传递/校验/单向流）、emit（触发/监听/emits 校验）。跨层级通信（provide/inject、状态管理 Pinia）、插槽 slot、动态组件/keep-alive 不在本章——它们是进阶，需要时查官方文档。
10 · 背景知识缺口

要完全吃透还差什么

现在就该补（阻碍写组件）：

  • 组件注册（全局 vs 局部）—— 怎么让一个组件在另一个组件里可用。本项目的 .vue + import 是局部注册。
  • 自定义事件的命名习惯—— 用 kebab-case（如 update-count），避免和原生事件冲突。

可以晚点补：

  • provide / inject—— 跨多层组件传数据，避免 props 逐层透传。
  • 插槽 slot—— 父组件往子组件里"塞内容"，比 props 传模板更灵活。
  • Pinia（状态管理）—— 多组件共享状态，大型项目必备。
